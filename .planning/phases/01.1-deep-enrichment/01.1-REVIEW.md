## Summary

Reviewed the deep enrichment pipeline additions: RSS episode extraction, company homepage scraping, Ollama LLM summarization, Airtable upload mapping, and pipeline orchestration. The code is generally clean, well-structured, and follows the existing patterns in the codebase. Key concerns are: (1) a silent failure path in web scraping that can mask connection errors vs. genuinely empty pages, (2) missing env var validation in upload.py that will produce a confusing runtime error, and (3) bare `except Exception` blocks that swallow useful diagnostic information.

## Critical Issues

### CR-01: Airtable upload passes None to API client when env vars are missing

**File:** `src/pipeline/upload.py:37-38`
**Issue:** `os.getenv()` returns `None` when the env var is not set. `Api(None)` and `api.table(None, None)` will produce a cryptic error deep in the pyairtable library rather than a clear "missing config" message. This is the only step that touches an external paid service, so a bad error here wastes debugging time.
**Fix:**
```python
def upload_to_airtable(prospects: list[ProspectDict]) -> dict:
    api_key = os.getenv("AIRTABLE_API_KEY")
    base_id = os.getenv("AIRTABLE_BASE_ID")
    table_id = os.getenv("AIRTABLE_PROSPECTS_TABLE_ID")

    missing = [name for name, val in [
        ("AIRTABLE_API_KEY", api_key),
        ("AIRTABLE_BASE_ID", base_id),
        ("AIRTABLE_PROSPECTS_TABLE_ID", table_id),
    ] if not val]
    if missing:
        raise RuntimeError(f"Missing required env vars: {', '.join(missing)}")

    api = Api(api_key)
    table = api.table(base_id, table_id)
    # ...rest unchanged
```

## Warnings

### WR-01: Bare except Exception silently swallows all errors in scrape_company_text

**File:** `src/pipeline/deep_enrich.py:109`
**Issue:** The `except Exception` block on line 109 catches everything (DNS failures, SSL errors, timeouts, encoding errors) and returns an empty string. While the comment references D-06 (prospect stays call-ready), this makes it impossible to distinguish "company has no meta description" from "network is down for all requests." If the network is down, every prospect silently gets empty company text and the only signal is the `scraped_fail` counter in the print output.
**Fix:** Log the exception type and URL so failures are diagnosable. Keep the non-blocking behavior:
```python
except Exception as exc:
    print(f"    Scrape failed for {url}: {type(exc).__name__}: {exc}")
    if path == "":
        continue
    return ""
```

### WR-02: Bare except Exception in call_ollama_deep hides LLM errors

**File:** `src/pipeline/deep_enrich.py:144`
**Issue:** Same pattern as WR-01. If Ollama is not running, every call silently returns None and the pipeline continues with degraded data. A `ConnectionRefusedError` on the first call should ideally short-circuit or at least warn loudly, since all subsequent calls will also fail.
**Fix:** At minimum, log the error. Optionally, detect connection errors on the first call and warn that Ollama may not be running:
```python
except Exception as exc:
    print(f"    Ollama deep enrich failed: {type(exc).__name__}: {exc}")
    return None
```

### WR-03: feedparser.parse silently returns empty feed on invalid URL or network error

**File:** `src/pipeline/deep_enrich.py:64`
**Issue:** `feedparser.parse()` never raises exceptions. It returns a feed object with an empty `entries` list and sets `feed.bozo` to 1 on parse errors or network failures. The code does not check `feed.bozo`, so a malformed RSS URL or DNS failure silently produces zero episodes with no warning.
**Fix:** Add a bozo check after parsing:
```python
feed = feedparser.parse(rss_url)
if feed.bozo:
    print(f"    RSS parse issue for {rss_url}: {feed.bozo_exception}")
entries = feed.entries
```

### WR-04: episode_details JSON in Airtable upload silently truncated at 10,000 chars

**File:** `src/pipeline/upload.py:29`
**Issue:** `json.dumps(...)[:10000]` can truncate mid-JSON, producing an invalid JSON string stored in Airtable. If a downstream consumer tries to parse this field, it will fail. Either truncate the list before serialization (fewer episodes) or store a note that data was truncated.
**Fix:**
```python
details_json = json.dumps(prospect.get("episode_details", []), ensure_ascii=False)
if len(details_json) > 10000:
    # Truncate to fewer episodes rather than cutting mid-JSON
    truncated = prospect.get("episode_details", [])[:2]
    details_json = json.dumps(truncated, ensure_ascii=False)
fields["episode_details"] = details_json
```

## Info

### IN-01: Module-level CONFIG load runs on import

**File:** `src/pipeline/deep_enrich.py:23`
**Issue:** `CONFIG = load_config()` executes at import time. If the JSON file is missing or malformed, any module that imports from `deep_enrich` will crash with an unhandled exception even if it only needs `ProspectDict` or `DeepEnrichResponse`. This is minor since the module is only imported when deep enrichment runs, but it couples import to filesystem state.
**Fix:** Consider lazy loading (load inside the function that uses it) or wrap in a try/except with a clear error message.

### IN-02: sys.path manipulation in run_pipeline.py

**File:** `src/run_pipeline.py:5`
**Issue:** `sys.path.insert(0, os.path.dirname(__file__))` is a common pattern for scripts but can cause subtle import issues if the module is also importable via the normal Python path. This is fine for a script entry point but worth noting.
**Fix:** No action needed for this project's scale. If the project grows, consider using a proper package install (`pip install -e .`).
