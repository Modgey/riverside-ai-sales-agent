## Summary

Phase 1.1 adds deep personalization data to call-ready prospects using zero-cost sources: RSS feeds already known from Phase 1, public company web pages, and the local Ollama instance already running for the qualify step. The three data products are: per-episode details (title, description, guest name) for the last 3 episodes; a company summary scraped from the prospect's domain; and a podcast theme summary generated by Ollama from the last 10 episode titles/descriptions.

The implementation follows patterns already established in the codebase. `feedparser` is already installed and the `parse_rss()` pattern is proven. `beautifulsoup4` is already installed (verified via pip list). Ollama is running with `gemma4:e4b` (the same model used in qualify.py). The Ollama + Pydantic JSON pattern from qualify.py transfers directly.

The key performance decision is to combine the two Ollama calls (theme summary + company summary refinement) into a single call per prospect. Live timing in this session showed: 2 separate calls = ~21s per prospect vs. 1 combined call = ~15s per prospect. At 15 call-ready prospects, that saves ~90 seconds of total runtime.

**Primary recommendation:** Build `src/pipeline/deep_enrich.py` following the qualify.py Ollama + Pydantic pattern. Use a combined single Ollama call per prospect for both theme summary and company summary. Cap description extraction at ~500 chars per episode. Homepage-only scraping is sufficient for most B2B SaaS sites; probe /about only if homepage meta description is absent.

### Pattern 1: Ollama Combined Call (follow qualify.py exactly)

```python
# Source: src/pipeline/qualify.py (verified in this session) + live timing test
class DeepEnrichResponse(BaseModel):
    podcast_themes: str
    company_summary: str

def call_ollama_deep(episode_titles: list[str], episode_descriptions: list[str], raw_company_text: str) -> DeepEnrichResponse | None:
    user_msg = json.dumps({
        "episode_titles": episode_titles,  # last 10 for themes
        "episode_descriptions": episode_descriptions,  # last 3 (truncated ~500 chars each)
        "company_raw": raw_company_text[:800],
    })
    try:
        response = ollama.chat(
            model=CONFIG["model"],
            messages=[
                {"role": "system", "content": CONFIG["system_prompt"]},
                {"role": "user", "content": user_msg},
            ],
            format="json",
            options={"temperature": CONFIG["temperature"]},
        )
        parsed = json.loads(response["message"]["content"])
        return DeepEnrichResponse(**parsed)
    except Exception:
        return None
```

### Pattern 2: RSS Re-parsing for Episode Details

Key findings from live testing on 2 real podcast feeds (fullratchet.net, codestory.co):

- `itunes_author` at entry level is almost always empty. The host is in `author` / `author_detail`, NOT the guest.
- Guest names reliably come from the episode **title** via regex heuristics.
- `summary` field contains HTML markup on some feeds. Strip with `re.sub(r'<[^>]+>', '', text)` or `BeautifulSoup(text, 'lxml').get_text()`.
- `content` field (list) may have cleaner text on some feeds but is inconsistent.

```python
# Source: verified against feedparser 6.0.12 output in this session
import re

GUEST_PATTERNS = [
    r'\bwith\s+([A-Z][a-z]+(?:\s+[A-Z][a-z]+){1,2})',
    r'\bft\.?\s+([A-Z][a-z]+(?:\s+[A-Z][a-z]+){1,2})',
    r'\bfeat(?:uring)?\s+([A-Z][a-z]+(?:\s+[A-Z][a-z]+){1,2})',
    r'\|\s*([A-Z][a-z]+(?:\s+[A-Z][a-z]+){1,2})',
]

def extract_guest_name(entry: dict) -> str | None:
    # itunes_author at entry level is almost always the host, not the guest -- skip it
    # Parse from title instead
    title = entry.get("title", "")
    for pattern in GUEST_PATTERNS:
        match = re.search(pattern, title)
        if match:
            return match.group(1)
    return None

def clean_description(raw: str, max_chars: int = 500) -> str:
    text = re.sub(r'<[^>]+>', ' ', raw)
    text = re.sub(r'\s+', ' ', text).strip()
    return text[:max_chars]

def extract_episode_details(rss_url: str) -> list[dict]:
    feed = feedparser.parse(rss_url)
    details = []
    for entry in feed.entries[:3]:
        details.append({
            "title": entry.get("title", ""),
            "description": clean_description(entry.get("summary", "")),
            "guest_name": extract_guest_name(entry),
            "published": entry.get("published", ""),
        })
    return details
```

### Pattern 3: Company Homepage Scraping

```python
# Source: verified with live requests to fullratchet.net and codestory.co in this session
import requests
from bs4 import BeautifulSoup

def scrape_company_text(domain: str, timeout: int = 8) -> str:
    url = f"https://{domain}"
    headers = {"User-Agent": "Mozilla/5.0"}
    try:
        resp = requests.get(url, timeout=timeout, headers=headers)
        resp.raise_for_status()
    except Exception:
        return ""
    soup = BeautifulSoup(resp.content, "lxml")
    # Priority: og:description > meta description > h1 text
    for selector in [
        ("meta", {"property": "og:description"}),
        ("meta", {"name": "description"}),
    ]:
        tag = soup.find(*selector)
        if tag and tag.get("content", "").strip():
            return tag["content"].strip()
    h1 = soup.find("h1")
    return h1.get_text(strip=True) if h1 else ""
```

Live test results: fullratchet.net returned 153-char meta description. codestory.co returned 81-char meta description. Both useful without needing /about page probe. [VERIFIED: live HTTP requests in this session]

### Anti-Patterns to Avoid

- **Caching RSS parse output from discover.py and re-using it:** discover.py only extracts `entries[:10]` titles for cadence, not episode descriptions. Re-parsing is necessary to get `summary` fields per D-01.
- **Using `itunes_author` at entry level for guest names:** Verified empty on 2/2 tested feeds. Use title regex heuristics instead.
- **Separate Ollama calls for theme and company summary:** Adds ~6s per prospect (21s vs 15s measured). Combined call is faster and simpler.
- **Calling ollama without json format mode:** Without `format="json"`, the model wraps output in markdown fences. Follows qualify.py's `format="json"` call.
- **Blocking the pipeline on a failed scrape:** D-06 is explicit: prospect stays call-ready with blank company_summary. No retries.
- **Passing raw HTML to Ollama:** Strip HTML before sending. Raw HTML with WordPress gutenberg blocks (as seen on fullratchet.net) will waste tokens and confuse the model.

## Common Pitfalls

### Pitfall 1: `itunes_author` at entry level is the host, not the guest

**What goes wrong:** Code reads `entry.get("itunes_author")` expecting guest name, gets the show host name (or empty string).

**Why it happens:** `itunes:author` in podcast RSS is defined at the feed level (show author) and sometimes repeated at entry level for the same host. It does not carry per-episode guest metadata.

**How to avoid:** Use title regex heuristics (patterns: "with [Name]", "ft.", "featuring", "| [Name]"). Return None if no match rather than falling back to `itunes_author`.

**Warning signs:** Guest name always equals the host name across all 3 episodes.

[VERIFIED: confirmed via live feedparser output on 2 feeds in this session]

### Pitfall 2: Ollama `format="json"` does not guarantee schema conformance

**What goes wrong:** Ollama returns valid JSON but with different key names than the Pydantic model expects (e.g., `"summary"` instead of `"company_summary"`). Pydantic raises a ValidationError.

**Why it happens:** The model interprets the task freely. JSON mode only ensures parseable JSON, not your schema.

**How to avoid:** System prompt must name the exact keys. E.g., "Return JSON with exactly these keys: podcast_themes (string), company_summary (string)." Wrap Pydantic parse in try/except and return None on failure (same pattern as qualify.py's `call_ollama` function).

**Warning signs:** Intermittent ValidationErrors, especially on prospects with unusual data.

[VERIFIED: live test showed model followed schema when explicitly named in system prompt]

### Pitfall 3: HTML in episode `summary` field

**What goes wrong:** The `summary` field contains WordPress Gutenberg block HTML comments and tags, producing garbled LLM input.

**Why it happens:** Many podcast hosting platforms (Libsyn, RedCircle) store the episode description as HTML. feedparser preserves it as-is.

**How to avoid:** Always strip HTML from `summary` before passing to Ollama or storing. `re.sub(r'<[^>]+>', ' ', text)` is sufficient; `BeautifulSoup` for thoroughness.

**Warning signs:** Ollama input contains `<!--`, `wp:paragraph`, `<p>` tags.

[VERIFIED: fullratchet.libsyn.com/rss summary field confirmed to contain HTML in this session]

### Pitfall 4: upload.py currently reads `score.json` hardcoded

**What goes wrong:** After deep_enrich.py runs and writes deep_enrich.json, the upload step still reads score.json (missing the new enrichment fields).

**Why it happens:** `run_pipeline.py:run_upload()` calls `load("score")` explicitly.

**How to avoid:** Two changes needed: (1) `run_pipeline.py` must add `run_deep_enrich()` between `run_score()` and `run_upload()` and update `run_upload()` to `load("deep_enrich")`; (2) `upload.py:prospect_to_airtable_fields()` must include the new fields.

**Warning signs:** Upload step runs fine but Airtable shows no episode_details/company_summary columns.

[VERIFIED: read run_pipeline.py and upload.py in this session]

### Pitfall 5: Ollama timing at demo scale

**What goes wrong:** 15 prospects x 15s per call = 3.75 minutes of silence. Demo or reviewer thinks it hung.

**Why it happens:** gemma4:e4b is a 9.6 GB model; inference takes ~12-15s per call on CPU/GPU.

**How to avoid:** Print per-prospect progress during the loop (same pattern as qualify.py's `print(f"  Qualifying... {i+1}/{total}")`). Show: "  Deep enriching {i+1}/{total}: {podcast_name}...".

[VERIFIED: live timing confirms ~15s per combined Ollama call in this session]

## Code Examples

### Minimal Deep Enrich Module Structure

```python
# src/pipeline/deep_enrich.py
# Mirrors the structure of qualify.py

import json, os, re
import feedparser
import ollama
import requests
from bs4 import BeautifulSoup
from pydantic import BaseModel

from pipeline.models import ProspectDict

CONFIG_PATH = os.path.join(os.path.dirname(__file__), "prompts", "deep_enrich.json")
CONFIG = json.load(open(CONFIG_PATH))


class DeepEnrichResponse(BaseModel):
    podcast_themes: str
    company_summary: str


def deep_enrich_prospects(prospects: list[ProspectDict]) -> list[ProspectDict]:
    """Run deep enrichment on call-ready prospects only. Returns full list."""
    call_ready = [p for p in prospects if p.get("status") == "call-ready"]
    others = [p for p in prospects if p.get("status") != "call-ready"]

    print(f"  Deep enriching {len(call_ready)} call-ready prospects ({len(others)} others skipped)")

    scraped_fail = 0
    llm_fail = 0
    for i, prospect in enumerate(call_ready):
        print(f"  Deep enriching {i+1}/{len(call_ready)}: {prospect.get('podcast_name', '')[:50]}")
        # 1. RSS re-parse
        prospect["episode_details"] = extract_episode_details(prospect.get("rss_url", ""))
        # 2. Company scrape
        raw_company = scrape_company_text(prospect.get("domain", ""))
        if not raw_company:
            scraped_fail += 1
        # 3. Ollama combined call
        episode_titles = [e["title"] for e in prospect.get("episode_details", [])]
        episode_descs = [e["description"] for e in prospect.get("episode_details", [])]
        result = call_ollama_deep(episode_titles, episode_descs, raw_company)
        if result:
            prospect["podcast_themes"] = result.podcast_themes
            prospect["company_summary"] = result.company_summary
        else:
            llm_fail += 1
            prospect["podcast_themes"] = None
            prospect["company_summary"] = raw_company[:300] if raw_company else None
        # 4. Rebuild richer cheat sheet
        prospect["cheat_sheet"] = assemble_deep_cheat_sheet(prospect)

    print(f"  Deep enrich complete: {len(call_ready)} processed, {scraped_fail} scrape failures, {llm_fail} LLM failures")
    return call_ready + others
```
